# Troubleshooting History — Ansible / Molecule CI

Дата: 2026-02-24

## Решено

| Ошибка | Причина | Решение |
|--------|---------|---------|
| Массовые `ansible-lint` нарушения на production profile | Роли и molecule-сценарии расходились с актуальными lint-правилами проекта | Прогонять production lint до CI и исправлять паттерны системно, а не точечно после падения |
| `ansible.builtin.pacman` не работает в converge/prepare | Pacman-модуль живет в `community.general`, а не в `ansible.builtin` | Использовать `community.general.pacman` и держать collection в requirements/CI install |
| `command-instead-of-module` на проверках, где модуль реально непригоден | lint не знает контекст, например broken/неподходящий module для конкретной проверки | Оставлять точечный `# noqa` только с понятной причиной, не глушить правило широко |
| `syntax-check[specific]: 'ansible.builtin.*' is not a valid attribute for a Play` | `ansible-lint` сканировал task-only YAML как playbook и ожидал `hosts:` | Не хранить task-only файлы там, где lint читает их как playbook; делать verify валидным playbook или настраивать lint scope |
| Handler не найден: `The requested handler ... was not found` | `notify` и `name` handler сравниваются case-sensitive | Использовать единый регистр и `listen:` для стабильного имени handler |
| `AnsibleUndefinedVariable` в standalone `verify.yml` | Verify запускается отдельным playbook, defaults роли автоматически не загружаются | Явно подключать нужные переменные через `vars_files` или inventory verify-сценария |
| Converge не загружает нужный data file | Тест запускает роль без inventory/group vars, поэтому данные оказываются пустыми | В molecule converge/verify явно подключать нужные `vars_files` или задавать минимальный test inventory |
| Ansible 2.20+ не парсит `.ini` inventory через `auto` | `ini` plugin не был явно включен | Добавлять `ini` в `enable_plugins` или мигрировать inventory на YAML |
| Тест отключает сервис через `*_enable_service: false` и скрывает реальное поведение | Molecule-сценарий меняет бизнес-контракт роли ради прохождения теста | Тест должен готовить окружение, а не отключать проверяемое поведение роли |
| Molecule подменяет устанавливаемый продукт mock-бинарником | Сценарий проверяет собственную заглушку и пропускает ошибки загрузки, распаковки, установки и запуска настоящего продукта | Если установка и запуск входят в контракт роли, использовать официальный артефакт и проверять реальное поведение; имитировать только внешние системы вне контракта роли |
| One-shot service всегда показывает `changed` при `state: started` | Запуск one-shot unit не является стабильным desired state | Проверять enabled/configured state и фактическое состояние через доменную команду, например `nft list tables` |
| Старые файлы остаются на тестовой VM после sync | SCP/rsync без удаления не убирает удаленные локально файлы | Синхронизация test target должна удалять stale files или начинаться с чистого состояния |
| Синхронизированные файлы получают слишком широкие права | Sync-инструмент выставляет `0666`/неправильный mode для конфигов | После sync нормализовать mode или настроить sync так, чтобы он сохранял безопасные права |
| Проверка `which`/`command -v` пропускает сломанный бинарь | Бинарь существует в PATH, но не запускается после обновления библиотек | Проверять исполняемость через реальный `--version`/smoke command |
| Assert по `.timer` не находит timer в `ansible_facts.services` | `service_facts` возвращает service units, но не timer units | Проверять timer через `systemctl is-enabled <unit>.timer` |
| `locale_gen ... not available on your system` | Тестовый образ не содержал locale data из-за `NoExtract`/минимального glibc | Исправлять контракт образа или prepare-сценарий: locale data должны существовать до converge |
| Standalone/default Molecule сценарий не видит роли | `ANSIBLE_ROLES_PATH` указывал на несуществующий каталог | В сценарии указывать путь к фактическому каталогу ролей относительно `MOLECULE_PROJECT_DIRECTORY` |
| `community.general.timezone` или другой collection module не найден | CI установил Ansible, но не установил collections | Устанавливать project requirements/collections до molecule run |
| CLI-команда возвращает `rc=0`, но не применяет ожидаемые данные | Был использован неверный режим CLI; команда тихо создавала пустой результат | Проверять реальный контракт команды через help/output и verify конечного состояния, а не только exit code |
| Неверный asset/download URL для архитектуры | Локальное имя архитектуры не совпадало с naming scheme релизных артефактов | Маппинг архитектур строить по фактическим release assets и валидировать наличие asset до скачивания |
| Зафиксированная версия скачиваемого ПО возвращает HTTP 404 | Указанной версии или ожидаемого release asset нет на CDN, хотя значение выглядит синтаксически корректным | До фиксации версии проверять существование всех используемых URL и checksum-файлов у официального источника |
| Сервис невозможно запустить после установки из архива | Архив предоставляет исполняемые файлы, но в отличие от distro-пакета не устанавливает unit или другое init-specific определение сервиса | Если контракт archive-install включает управление сервисом, роль должна декларативно установить его определение для поддерживаемой init system |
| Проверка `stat.stat.executable` ломается или недоступна | Поведение `stat`/атрибутов зависит от версии Ansible и окружения | Для исполняемости проверять mode/permissions или явный `test -x` с controlled failure |
| `name[play]` на файле с `import_playbook` | `import_playbook` не является play, `name:` не удовлетворяет правилу lint | Для intentional wrapper-файла использовать точечный `# noqa: name[play]` |
| Переменные из `molecule.yml` не попали в Docker converge | Inventory/group vars сценария не совпали с фактическим execution path | Критичные test vars задавать в shared converge/vars или в проверенном inventory path сценария |
| Jinja condition падает на `regex_search()` как на не-bool значение | В новых версиях Ansible/Jinja строка из `regex_search` не является boolean assertion | Для boolean-проверок использовать Jinja test `is search(...)` или явное сравнение с `none` |
| JSON type assert не принимает строковое поле после `from_json` | `from_json` возвращает обычный `str`, а не `AnsibleUnsafeText` | В type assertions учитывать реальные типы после парсинга JSON |
| Verify видит старое состояние сервиса после изменения конфига | Handler выполняется в конце play, а verify запускается раньше | Перед verify добавлять `meta: flush_handlers`, если verify зависит от handler-applied state |
| `systemctl start` завершился успешно, но сервис не готов к работе | Для `Type=simple` systemd может вернуть success после запуска процесса, хотя приложение сразу после этого упало или еще не создало runtime socket/API | Проверять runtime-контракт доменным клиентом или smoke command, например `*-client ping/status`, а не только `systemctl` state |
| Idempotence ломается на директории, созданной systemd | `LogsDirectoryMode`/`StateDirectoryMode` в unit-файле перезаписывает mode после старта сервиса | Согласовать Ansible file mode с systemd unit mode или передать нужный mode через drop-in |
| AUR установка требует sudo, но пользователь сборки не может его получить | AUR helper запускается под отдельным пользователем, а пароль/askpass не передан | Передавать sudo password через безопасный runtime secret/askpass; не использовать NOPASSWD как обход |
| AUR пакет конфликтует с official package | Пакеты предоставляют один и тот же binary/file set | До установки AUR пакета явно удалять/конфигурировать конфликтующие official packages |
| Pacman package replacement ломает зависимости | Раздельное удаление старого пакета перед установкой replacement package может временно сломать reverse dependencies, например `iptables` -> `iptables-nft` и `iproute2/libxtables` | Делать замену атомарной через package-manager conflict prompt, например `community.general.pacman` с `extra_args: --ask=4`, а не двумя отдельными операциями remove/install |
| Ошибка проявилась только в изолированном CI-сценарии | Полный локальный playbook маскировал зависимость от соседних ролей или окружения | Тестировать сценарий изолированно и не полагаться на внешний playbook-контекст |
| После исправления первой причины CI падает на следующей ошибке того же сценария | Несколько независимых дефектов находились в одной цепочке, и более ранний сбой не позволял выполнить код с последующим дефектом | После каждого исправления заново проходить полный сценарий и считать причиной только подтверждённый факт; не считать первый найденный дефект единственным без повторного запуска |
| Docker prepare вызывает пакетный менеджер не той ОС | В multi-platform scenario задача package manager шла без `when`, а `gather_facts: false` не давал определить `os_family` | В prepare включать `gather_facts: true` и разделять package-manager tasks по OS family |
| Локальный Molecule fallback image расходится с образом CI | CI передаёт image через environment, а fallback в `molecule.yml` остаётся старым | Обновлять fallback и CI image reference вместе; оба пути должны использовать один контракт образа |
| Molecule platform не запускается без обязательной image-переменной | Reusable workflow не передал ожидаемую переменную окружения, а scenario не имеет рабочего fallback image | Передавать image-переменные через каждый уровень reusable workflow и сохранять проверенный fallback в `molecule.yml` |
| Docker-сценарий требует systemd, cgroups или привилегированный runtime | Обычный контейнер с `sleep infinity` не предоставляет service manager и нужные kernel interfaces | Если контракт может работать с host kernel, явно подготовить privileged systemd-контейнер, cgroups и runtime mounts; VM оставить для проверки полноценной kernel/host границы |
| Prepare заранее создаёт желаемое состояние роли | Converge становится no-op и тест не доказывает коррекцию состояния | Prepare должен создавать только внешние prerequisites, которыми роль не владеет |
| Сервис в контейнере ломается из-за systemd sandboxing | Container namespaces отличаются от VM и unit restrictions блокируют необходимый runtime path | Если runtime в Docker действительно нужен, готовить test-only unit drop-in в prepare; production роль не менять ради контейнера |
| Контейнеры тестовой VM не имеют сети | Host firewall блокирует forwarding Docker bridge | Исправлять host test environment и его bridge-forwarding policy, не роль приложения внутри контейнера |
| Base image перестаёт собираться после обновления rolling repository | Зеркала, package names или состав base repository изменились, а image contract это не обнаружил | Проверять обязательные пакеты при сборке образа, использовать надёжные зеркала и обновлять image mapping по фактическим репозиториям |
| Одна из поддерживаемых ОС отсутствует в Molecule matrix | Сценарий содержит только исходную платформу и поэтому скрывает distro-specific ошибки prepare/converge | Для кроссплатформенной роли включать каждую заявленную CI-платформу и давать ей собственные package prerequisites |
| `EBUSY` или `EXDEV` при изменении системного файла в контейнере | Runtime bind-монтирует файл, а Ansible пытается выполнить atomic rename через границу mount point | Не менять runtime-owned bind mount как обычный файл; для test-only случая убрать mount в prepare либо выбрать поддержанный интерфейс настройки |
| Проверочная команда отсутствует в minimal image | Тест полагается на legacy utility, не входящую в контракт роли или базового образа | Использовать уже гарантированный системный интерфейс либо явно установить utility как test prerequisite |
| HTTPS или TLS-сервис падает с `CERTIFICATE_VERIFY_FAILED` | Minimal image не содержит актуальный CA bundle | Устанавливать и обновлять `ca-certificates` в prepare до сетевых операций; не отключать TLS verification |
| File logging не срабатывает после ошибки в контейнере | Код сначала обращается к отсутствующему `/dev/log`, исключение прерывает запись в основной log file | Сначала записывать обязательный file log, затем best-effort отправлять сообщение во внешний syslog transport |
| Prepare устанавливает prerequisite только на одной платформе | OS-specific package task существует для первой distro, а остальные ветки случайно зависят от состава base image | Явно сопоставлять prerequisite packages каждой поддержанной OS family и собирать facts до выбора package manager |
| Hardware или kernel-module path проходит в generic Docker CI | Контейнер использует kernel host и не имеет реального PCI/GPU/device контекста тестируемой системы | Не заявлять hardware coverage; kernel modules, DKMS, initramfs и device runtime проверять в подходящей VM или bare-metal среде |
| Package task успешен, но post-install hook сломан | Package manager установил файлы, однако отдельная DKMS/module generation операция завершилась нерабочим результатом | Для пакетов с kernel hooks проверять конечный module/build contract в среде с соответствующим kernel и headers |
| Service restart сразу после package install нестабилен | Package post-install сам запустил сервис, а следующая задача немедленно пытается перезапустить ещё не завершивший инициализацию процесс | Prepare должен завершить внешнюю package initialization до converge; роль затем применяет конфигурацию и обычное desired service state |
| Первый запуск сервиса требует созданных им системных пользователей или subordinate ID ranges | Конфигурация включает namespace/remapping до того, как package-provided initialization создала prerequisite state | Package layer или test prepare должен выполнить документированный first-start prerequisite; роль не должна скрывать его через `ignore_errors` |
| Nested daemon не запускается с выбранным storage backend | Внутренний daemon пытается использовать storage поверх несовместимого backing filesystem контейнера | Давать nested daemon отдельный `/var/lib` mount с подходящей файловой системой либо использовать backend, поддержанный именно тестовой средой |
| Daemon отвергает options, допустимые только для другого backend | Шаблон безусловно передаёт backend-specific параметры, например rotation options неподдерживающему logging driver | Рендерить option block только для совместимых backend значений и валидировать итоговый config самим daemon до deploy |
| Initramfs rebuild запускается в generic Docker scenario | Контейнер использует host kernel и не имеет собственного полноценного modules tree, поэтому rebuild бессмысленен или падает | Проверять initramfs configuration в контейнере, а сам rebuild и полученный image проверять в VM или bare-metal среде |
| Hardware runtime command запускается без устройства или display session | Утилите нужен реальный GPU, kernel module, device node или пользовательская графическая сессия, которых generic CI container не предоставляет | Ограничить Docker package/config/idempotence контрактом; runtime hardware behavior проверять в подходящей VM или bare-metal scenario |
| Vagrant падает с `No usable default provider` после cache hit | `vagrant-libvirt` восстановлен из кэша с native `.so`, собранным против другой версии `libvirt-dev` | Включать версию системной библиотеки (`libvirt-dev`) в cache key для vagrant gems |
| `vagrant plugin repair` завершается успешно, но provider не появляется | `gem pristine` может не перекомпилировать native extension, если исходный `.gem` недоступен в локальном cache | Не считать `vagrant plugin repair` надежным fix; делать cache miss/force reinstall с правильным cache key |
| Molecule vagrant scenario отличается от рабочего шаблона ролей | Унаследованы старые `inventory.host_vars.localhost` или отсутствуют стандартные provisioner options | Сверять новые сценарии с актуальной reference-role и держать `skip-tags: report` там, где report ломает тестовый вывод |
| Регулярное выражение детекции GPU перевернуто относительно `lspci -nn` | Реальный вывод идет как `VGA compatible controller: NVIDIA Corporation ...`, то есть класс устройства перед vendor, а regex вида `NVIDIA.*(?:VGA\|Display)` ожидает vendor перед классом и никогда не матчится | Строить детектор по фактическому stdout: `(?:VGA\|Display).*NVIDIA`, `(?:VGA\|Display).*(?:AMD\|ATI)`, `(?:VGA\|Display).*Intel Corporation`; покрывать детекцию реальными примерами `lspci` |
| `lspci` вызывается без гарантированного `pciutils` | На минимальной системе `lspci` может отсутствовать, а `failed_when: false` превращает ошибку в "GPU не обнаружена" без явного падения | До авто-детекции валидировать или устанавливать `pciutils`; отсутствие инструмента детекции должно быть отдельной ошибкой, а не результатом "GPU нет" |
| `failed_when: false` маскирует ошибку команды детекции | Сбой команды чтения состояния выглядит как допустимое отсутствие найденного устройства | Разделять "команда не выполнилась" и "команда выполнилась, но объект не найден"; для допустимых rc задавать точный `failed_when` |
| `register` и использование переменной отличаются префиксом `_` | Задача пишет `<role>_dracut_check`, а выражение читает `_<role>_dracut_check`; на Ubuntu/Debian это дает undefined variable | Имя в `register` и всех местах чтения должно совпадать; проверять все distro/init ветки |
| `Undefined variable` скрыта коротким замыканием на Arch path | Выражение выбирает `mkinitcpio` раньше чтения несуществующей переменной, поэтому Arch проходит, а Debian/Ubuntu падает | Не считать успешную ветку с коротким замыканием доказательством корректности всего выражения; тестировать каждую ветку выбора |
| Как описано в разделе 1: `template` использует `_<role>_has_*`, а tasks выставляют `<role>_has_*` | Jinja2 воспринимает несуществующую `_<role>_has_*` как false; отрендеренный файл существует, но в нем нет ожидаемого содержимого, например `LIBVA_DRIVER_NAME=` | Держать одинаковые имена фактов на записи и чтении; проверять содержимое отрендеренного файла, а не только его существование |
| `LIBVA_DRIVER_NAME` не попадает в файл переменных окружения | `template` рендерит только comments из-за несовпадения имен facts; VA-API hardware acceleration не получает ожидаемую переменную окружения | Проверять конкретные строки в отрендеренном файле переменных окружения для NVIDIA/AMD/Intel path |
| Принудительное переопределение вида `vendor: intel` скрывает ветку авто-детекции | CI принудительно идет по pure Mesa path без `lspci`, DKMS и kernel modules; auto-detection и bare-metal branch остаются непроверенными | В описании сценария явно фиксировать, что принудительный vendor проверяет тестовую ветку, а auto-detect требует отдельной проверки с реальным stdout/device |
| Сценарий standalone/default меняет поведение локальной детекции host | Shared converge принудительно выбирает тестовый vendor вместо фактической GPU host machine | Либо документировать расхождение default scenario, либо держать отдельный scenario для real-host auto-detect |
| Verify проверяет отсутствие NVIDIA artifacts в non-NVIDIA path как основной контракт | Тест фокусируется на `state: absent` для `/etc/modprobe.d/nvidia.conf`, blacklist и initramfs drop-ins, хотя роль может быть additive и не удалять ранее созданные файлы | Verify должен проверять заявленный контракт текущей роли; отсутствие файлов проверять только если роль действительно владеет cleanup/absent behavior |
| Debian install branch является `debug` placeholder | Пользователь на Debian/Ubuntu не получает драйверы и не получает явный fail, если branch выглядит поддержанным | Нереализованная ветка должна явно `fail` с понятным сообщением или быть документирована как unsupported/stub |
| `Package not found` из-за неверного имени пакета для Ubuntu 24.04 | Имя пакета в роли или verify не совпадает с фактическим именем в noble repos, например `intel-media-va-driver`, `vulkan-tools`, `mesa-vulkan-drivers` | Проверять маппинг пакетов на целевой версии дистрибутива и обновлять роль и verify одновременно |
| Idempotence падает из-за `template`/`file` task | Task каждый прогон меняет файл или права, поэтому второй converge показывает changed вместо ok | Сравнивать desired state с текущим состоянием, фиксировать стабильный content/mode/owner/group и проверять второй converge |
| `Service not found` на vendor-specific service | Service task пытается управлять NVIDIA service на non-NVIDIA path или до установки пакета, который предоставляет unit | Service logic должна быть привязана к фактически настроенному vendor и наличию unit; отсутствие unit не должно маскироваться как успешная настройка |
| Manual vendor override пропускает detector prerequisite | При ручном vendor preflight может не требовать `pciutils`, поэтому сценарий проходит без инструмента, нужного для auto-detect path | Отдельно проверять auto-detect path с `pciutils` и явно описывать, что manual override не покрывает detector prerequisites |
| `HibernateMode=hibernate` не является допустимым значением `sleep.conf(5)` | `HibernateMode=` принимает `platform`, `shutdown`, `reboot`, `suspend`, `test_resume`; значение `hibernate` является user action, а не mode | Валидировать enum defaults по man page целевого конфига; заменить default на допустимое значение |
| DMI/laptop fact может стать строкой `"False"` | `set_fact` через folded scalar и `{{ }}` может дать строку; непустая строка `"False"` truthy в Jinja2 | Приводить условия к bool (`| bool`) или писать выражение так, чтобы сохранялся native boolean |
| `cpupower frequency-set` ломает idempotence | Task всегда сообщает changed через `changed_when: true` и скрывает ошибки через `failed_when: false` | Читать текущий governor через `cpupower frequency-info -p`, сравнивать с целевым и запускать `frequency-set` только при несовпадении |
| `multilib: true` требует `[multilib]` repo | Установка `lib32-*` packages падает с `target not found`, если `[multilib]` не включен в `/etc/pacman.conf` | Проверять `[multilib]` в validate/preflight или готовить repo в сценарии, где тестируется multilib |
| `kernel.unprivileged_userns_clone` может отсутствовать на kernel без distro patch | Key является Arch/Debian patch, не upstream Linux; `sysctl --system` может дать warning/noise | Проверять поддержку sysctl key перед рендером или применением либо вынести в platform-specific набор |
| Molecule verify хардкодит sysctl expected values | Проверки вида `!= 10`, `!= 524288`, `!= 4096` падают при пользовательских overrides через group_vars | Verify должен использовать role variables (`{{ sysctl_vm_swappiness }}` и аналоги), а не литералы |
| `xf86-video-nouveau` может быть deprecated или удален из Arch repos | Package помечен как устаревший в пользу встроенного modesetting DDX и может исчезнуть из repo | Проверять package availability на актуальной VM/CI и обновлять package mapping |
| TLP config template содержит только 7 параметров | Реальный TLP config имеет ~100+ опций; полный template deploy `/etc/tlp.conf` теряет upstream defaults | Использовать targeted line management или включать полный upstream config как базу, если роль владеет всем файлом |
| Desktop governor не проверяется в verify | Verify проверяет package и `sleep.conf`, но не проверяет фактически установленный governor `schedutil` | Добавить runtime check вроде `cpupower frequency-info -p` в подходящей среде |
| Template task не использует `validate:` перед deploy | Невалидный generated config может быть записан до проверки и сломать применение | Использовать `validate:` (`sysctl -p %s` и аналоги) или отдельную pre-deploy validation |
| Runtime diagnostics лежит в общем verify и использует init-specific команды | Общий verify вызывает `journalctl`/`systemctl` даже для ролей, где заявлены разные init systems | Диагностику падений разделять по init-specific файлам; общий verify должен собирать только контрактные проверки и подключать диагностику подходящего init system |
| Log polling backend падает или ничего не читает в тесте | Роль или тест выбрали backend, который читает обычный log-файл, но minimal image/VM этот файл не создает | Если сценарий использует file-based polling backend, создавать fixture log file в `prepare.yml`; не перекладывать создание чужого log source в роль |
| Автовыбор log backend делает VM/CI сценарий недетерминированным | `backend=auto` может выбрать другой источник логов в зависимости от установленных Python-модулей, journald/syslog и состава образа | В тестовом сценарии фиксировать backend, соответствующий подготовленному log source; default роли документировать отдельно от test override |
| Yamllint line-length warnings остаются после успешного syntax/lint | Длинные Jinja2 expressions и molecule vars_files paths дают non-blocking noise | Разбить длинные строки или настроить `.yamllint`, чтобы предупреждения не скрывали реальные ошибки |
| Converge падает на отсутствующей группе, директории или binary внешнего компонента | Изолированный test image не предоставляет предпосылку роли, которую полный workstation pipeline обычно создаёт раньше | Декларативно готовить внешнюю предпосылку в scenario `prepare.yml`; не переносить чужую установку в тестируемую роль и не отключать её контракт |
| После deploy sudoers следующие задачи теряют `become` | Новая групповая policy требует пароль и перекрывает более раннее NOPASSWD-правило automation user | В test environment сохранять узкое NOPASSWD-правило только для automation user и загружать его после управляемой групповой policy |
| `command.argv` передаёт `[None]` вместо аргумента `-` | Bare YAML-элемент `- -` разбирается как вложенный пустой list item, то есть `null`, а не строковый дефис | В `argv` всегда записывать отдельный дефис как quoted string: `- "-"` |
| Scenario variable не применяется после подключения `vars_files` | Play-level `vars_files` имеет более высокий precedence и перекрывает inventory/group vars сценария | Не дублировать одно решение на разных уровнях; передавать test decision через один заранее выбранный source с понятным precedence |
| Scenario variable не применяется после `include_vars` внутри роли | Загруженная задачей переменная имеет более высокий precedence, чем inventory `host_vars`, и перекрывает значение сценария | Внешний контракт держать в `defaults/`; через `include_vars` загружать только внутренние значения, которые оператор и тесты не должны переопределять |
| Prepare пишет fixture в package-owned directory до установки пакета | `prepare` выполняется до `converge`, поэтому директории, создаваемые пакетом роли, могут отсутствовать в minimal Docker image | Не считать package-owned директории существующими в prepare; если fixture действительно нужен до converge, явно создать parent directory в prepare или перенести требование в роль |
| Regex с `\b` в Jinja2/Ansible ищет backspace вместо word boundary | Jinja2 строковый литерал превращает `\b` в ASCII backspace до передачи строки в regex engine | Для word boundary использовать `\\b`; для простых проверок предпочитать структурные/effective checks без regex |
| Broad `sed` rewrite ломает YAML/Ansible файлы | Массовая замена меняет не только нужное выражение, но и task names, сообщения, комментарии или соседние YAML-значения | Делать целевые правки по уникальному контексту и проверять diff; для структурных файлов не использовать широкие regex-rewrite команды |
