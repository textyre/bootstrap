# Ansible плох? Нет, просто готовьте его правильно!

> Конспект доклада Сергея Печенко (Райффайзен Банк) на DevOpsConf 2022.
> Источник: [YouTube](https://www.youtube.com/watch?v=E3zGVZB0XiA) | ~46 минут | 6.8k просмотров

**Контекст:** Платформенная команда предоставляет GitLab as a Service на ~2000 пользователей, управляя всем через Ansible.

---

## Детальный обзор

Доклад формулирует **7 правил хорошего тона** при работе с Ansible, нарушение которых ведёт к техдолгу и «кризису автоматизации» — ситуации, когда никто не верит, что автоматизация работает правильно. Стоп-слово для этих правил — **кризис**.

Ключевая мысль: Ansible прощает слишком многое и дружелюбен к плохим практикам. Чтобы сделать из него хороший инструмент, нужно следовать дисциплине, использовать декларативность вместо императивного кода и не изобретать велосипеды.

---

## Правило 1: Шаблоны — не для структурированных форматов

Не используйте Jinja2-шаблоны для генерации YAML-файлов как текста.

**Почему плохо:**
- Ansible не знает про структуру YAML в шаблоне — для него это просто набор букв. Переиспользование переменных невозможно
- Вставка сложных структур (словарей, списков) ломает формат — обязательно в самый неподходящий момент
- При изменении дефолтов нужно менять и переменные, и шаблон — два места для ошибки
- Человеку всё равно придётся читать и понимать этот YAML

**Как правильно:** Используйте фильтр `to_yaml` / `to_nice_yaml` — пусть инструмент сам форматирует. В шаблонах доступны «задачки со звёздочкой»: можно создавать временные переменные в операторных скобках `{% set ... %}`, которые живут только на момент шаблонизации файла.

---

## Правило 2: Play = декларативная точка встречи

Не пишите `hosts: all` с условиями `when` для выбора нужных групп.

**Почему плохо:**
- `ansible-lint` не проверяет `when`-условия — корректность проверяется только в runtime
- Ошибка в условии или `--limit` превращает ваш деплой «в тыкву»
- Чтобы прочитать такой плейбук, нужно «запустить Jinja в голове»

**Как правильно:** Используйте модуль `group_by` для создания динамических групп по значению переменной. Потом в следующих plays работайте с этими группами. Плюсы:
- Плейбук становится **декларативным**
- Если переменная пуста — группы пустые — Ansible не делает ничего (fail-safe)

---

## Правило 3: Роли — портативные компоненты

Роль = портативная колонка с ручкой. Стандартные входы (параметры), понятные выходы, переносимость.

**Почему плохо затаскивать в роль имена групп/хостов:**
- Теряется переносимость — роль привязана к конкретному инвентарю
- Невозможно тестировать
- Сложность растёт без пользы

**Как правильно:** Всё, что относится к инвентарю, передавайте через **параметры роли** (defaults/vars). Описывайте параметры в `README.md` — любой человек в компании сможет понять интерфейс роли. **Одна роль = один компонент.**

---

## Правило 4: Декларативность через переменные

Переменные должны быть описаны в проекте через `group_vars`, `host_vars`, `inventory_vars`.

В Ansible 22 уровня приоритета переменных, но на практике достаточно **4-х:**
1. `role defaults` (самый низкий приоритет)
2. `inventory_vars`
3. `group_vars`
4. `host_vars`

Декларативность помогает не писать лишний код и не переписывать куски Ansible на самом Ansible.

---

## Правило 5: Обработка событий — используйте `listen`

Вместо вызова хендлеров по имени (`notify: restart nginx`) используйте **topics** через `listen`.

**Преимущества:**
- Один `notify` может запустить **несколько хендлеров** сразу
- Хендлеры могут быть сложнее одного task'а — цепочки задач
- **Кросс-ролевые notify**: роль приложения делает `notify: restart nginx`, а роль nginx слушает этот топик и выполняет рестарт. Принцип: одна роль — один компонент, роль приложения НЕ настраивает nginx

---

## Правило 6: Python-зависимости — будьте сопротивлением

Модули Ansible часто тянут Python-зависимости (`psycopg2`, Kubernetes-клиент и т.д.), которые:
- Поставляются как бинарные пакеты неизвестного происхождения
- Почти всегда протухшие и сложные в установке
- Никто не сканирует двоичные куски в питоновых зависимостях

**Как правильно:** Ansible имеет встроенный загрузчик модулей — можно положить модуль прямо в проект. Пишите модули на **чистом Python** без внешних зависимостей. Спикер написал модуль для PostgreSQL, который просто кидаешь в проект — и он работает (статья на Хабре).

---

## Правило 7: Не управляйте пользователями через Ansible

«LDAP-солдат месяца» — мем из Ansible-чата в Telegram. Минимум раз в месяц кто-то приходит с желанием раскидать SSH-ключи и пользователей через Ansible.

**Почему плохо:**
- Получаются **локальные** пользователи — между хостами это разные сущности
- Нет сквозных идентификаторов, сквозного отключения, SSO
- Уволенного сотрудника нужно прокатывать скриптом по всем хостам

**Используйте:** Active Directory, FreeIPA, LDAP (RFC 1487, 1993 год).

---

## Бонус: Не катайте Ansible на контейнеры

- Нужно ставить ту же пачку невменяемых Python-зависимостей
- Нет планировщика/кластера — никто не будет пинать и перезапускать контейнеры
- Не масштабируется (нет auto-scaling, scheduling)
- Docker-compose через Ansible — тоже не вариант

**Используйте Kubernetes** — он даёт service discovery, балансировку, scheduling.

---

## Контрольный вопрос (ключевой takeaway)

> **«Конфигурация чего именно описана в этом task'е?»**

Если вы не можете ответить на этот вопрос — значит, task нужно переписать. Это проверка на идемпотентность и чистоту дизайна.

## Дополнительные темы из Q&A

- **host_vars** — хороши как временный костыль, потом лучше создать группу с одним хостом и перейти на group_vars
- **inventory_vars** — подходят для глобальных настроек (SSH-подключение и пр.), которые не меняются между хостами
- **Неопределённые дефолты** — лучше сделать параметр обязательным (без дефолта) и валидировать через `assert`, чем скрывать неочевидные значения
- **register + when** вместо хендлеров — bad practice, не проходит проверку контрольным вопросом
- **mitogen** — ускоряет, но слишком много меняет под капотом; Ansible сам умеет ускоряться (статья на Хабре)
- **AWX** — не обязателен для масштабирования. У них в банке несколько тысяч хостов без AWX
- **Execution Environments** от Red Hat — «ребята продают свой продукт» (Ansible Automation Platform), ползучий vendor lock-in
- **Гибридная структура каталогов**: `group_vars/all` в корне проекта читается для всех inventory
- **Изучайте Ansible по документации**, а не по русскоязычным YouTube-видео (которые часто снимают люди, только разбирающиеся с инструментом)

## Чеклист: Actionable Items

### Шаблоны
- [ ] Заменить Jinja2-шаблоны для YAML на фильтры `to_yaml` / `to_nice_yaml`
- [ ] Использовать `{% set %}` для временных переменных в шаблонах вместо глобальных переменных
- [ ] Проверить все `.j2` шаблоны — нет ли среди них генерации YAML как текста

### Плейбуки и группы
- [ ] Убрать `hosts: all` + `when` из плейбуков
- [ ] Перейти на `group_by` для динамического создания групп
- [ ] Сделать play декларативными: `hosts: <конкретная_группа>` + роли/таски

### Роли
- [ ] Вынести все ссылки на инвентарь (имена групп, хостов) из ролей в параметры
- [ ] Описать параметры каждой роли в `README.md`
- [ ] Применить принцип «одна роль = один компонент»
- [ ] Проверить, что роли переносимы между проектами/инвентарями

### Переменные и декларативность
- [ ] Переместить переменные в `group_vars` / `host_vars` / `inventory_vars`
- [ ] Перевести inventory из INI-формата в YAML (INI не поддерживает словари)
- [ ] Использовать не более 4 уровней приоритета: role defaults, inventory_vars, group_vars, host_vars
- [ ] Сделать важные параметры обязательными (без дефолтов) с валидацией через `assert`
- [ ] Префиксовать переменные ролей именем роли для избежания коллизий

### Хендлеры
- [ ] Перейти с `notify: <handler_name>` на `listen: <topic_name>`
- [ ] Реализовать кросс-ролевые notify через topics
- [ ] Убрать паттерн `register` + `when` в пользу хендлеров

### Зависимости
- [ ] Проверить все Python-зависимости модулей Ansible в проекте
- [ ] Заменить модули с бинарными зависимостями на чисто-питоновые альтернативы
- [ ] Использовать встроенный загрузчик модулей Ansible (library/ в проекте)

### Архитектура
- [ ] Не управлять пользователями через Ansible — внедрить LDAP/FreeIPA/AD
- [ ] Не управлять контейнерами через Ansible — использовать Kubernetes
- [ ] Для каждого task'а задавать контрольный вопрос: «Конфигурация чего описана?»
- [ ] Использовать `ansible-inventory` для автоматического тестирования инвентаря (вывод JSON и сравнение с эталоном)
