## Обзор

Доклад о практическом опыте построения системы сбора, обработки и визуализации логов
в высоконагруженном проекте (маркетплейс Юла). Олег рассказывает путь от единственного
сервера с базовым мониторингом до инфраструктуры с сотнями сервисов, Kubernetes и
выделенной командой поддержки. Основной фокус — на стеке ELK (Filebeat + Logstash +
Elasticsearch + Kibana/Grafana), подводных камнях при высоких нагрузках и способах
их решения.

---

## 1. Эволюция: от одного сервера к маркетплейсу

- **Начало:** маленький интернет-магазин, один сервер (LAMP), ~1k RPM.
  - Базовый мониторинг: host-tracker (доступность), Zabbix (железные метрики), аналитика (баннеры).
  - Проблемы решались простым ребутом; даунтайм в 6 часов никого не беспокоил.
- **Рост:** маркетплейс с высоким RPS, Kubernetes + монолиты на физических серверах,
  множество БД, команда саппорта.
  - «Сайт не работает» — уже не пятисотка, а сломанный функционал, который базовый мониторинг не ловит.
  - Grep по SSH на 100+ сервисах — нереально.

---

## 2. Что делать с логами

1. **Централизованно собирать** логи со всей инфраструктуры (серверы, K8s, БД, скрипты).
2. **Обрабатывать и нормализовать** — «причесать» сырые stacktrace'ы и JSON в читаемый вид.
3. **Дать доступ саппорту** — чтобы собрать контекст по кейсу пользователя и оформить баг.
4. **Дать доступ разработке** — чтобы быстро воспроизвести и пофиксить баг.
5. **Строить мониторинг:**
   - **Трендовый** (Grafana) — тенденции, графики.
   - **Оперативный** (Zabbix) — алертинг в реальном времени.

---

## 3. Инструменты стека

| Задача | Инструмент | Почему |
|--------|-----------|--------|
| Сбор логов | **Filebeat** | Прост, хорошая документация, быстрый, нетребователен к ресурсам |
| Обработка / обогащение | **Logstash** | Большой набор плагинов, большое комьюнити, масштабируется |
| Хранение | **Elasticsearch** | Быстрый, полнотекстовый поиск, масштабируется, мощный API |
| Поиск / визуализация логов | **Kibana** | Удобные фильтры, полнотекстовый поиск, дашборды, API |
| Трендовая визуализация | **Grafana** | Красивые графики, интеграция со всеми хранилищами, 100+ плагинов, алертинг из коробки |

### Базовая схема

```
Filebeat → Logstash → Elasticsearch → Kibana / Grafana / Zabbix
```

### Highload-схема

```
Filebeat → Kafka (очереди) → Logstash (с балансировкой) → Elasticsearch (кластер) → Kibana / Grafana
```

## 4. Подводные камни

### 4.1 Filebeat и log rotate

- **Проблема:** после `logrotate` Filebeat продолжает держать дескриптор удалённого файла — место на диске не освобождается.
- **Решение:** три параметра в конфиге Filebeat — закрытие неактивных, удалённых и переименованных файлов.

### 4.2 Logstash — бутылочное горлышко

- Logstash работает **пачками** (batch, по умолчанию ~125 записей).
- Pipeline: `main input → filter → output pipelines`.
- **Проблема:** при росте нагрузки один входящий pipeline (из Kafka) становится бутылочным горлышком — output-пайплайны простаивают.
- **Решение:** разделить входящие pipeline по топикам Kafka — отдельный input на каждый топик.
- Эта схема работоспособна до **~700 ГБ/сутки** логов.

### 4.3 Elasticsearch — шарды и GC

- Индекс дробится на **шарды** (primary + replica), распределённые по нодам.
- **Формула:** не более **20 шардов на 1 ГБ heap** (при 4 ГБ heap → max 80 шардов на ноду).
- **Проблема:** при превышении лимита шардов потребление heap растёт → при ~75% триггерится **Stop-the-World GC** → нода замораживается, не отвечает на запросы, может вывалиться из кластера по таймауту.
- **Решение:**
  - Придерживаться формулы 20 шардов / 1 ГБ heap.
  - Группировать логи в индексы по типу данных, не раздувать количество индексов.

### 4.4 Kibana — утечка памяти Node.js

- Наблюдалась утечка памяти: потребление растёт «лесенкой» до лимита, затем instance падает.
- Попытки тюнить GC не помогли.
- **Workaround:** выделить в 2 раза больше памяти, чем пиковое потребление.

### 4.5 Grafana

- Практически без нареканий. Единственный замеченный недостаток — нельзя было создавать папки для дашбордов (исправлено в версиях после 6.x).

## 5. Альтернативные инструменты

| Инструмент | Комментарий |
|-----------|------------|
| **Fluentbit** | Хорош при малых объёмах; при росте требует тонкой настройки (чанки, буферы) |
| **Fluentd** (вместо Logstash) | Те же проблемы при росте нагрузки |
| **Graylog** | Не пробовали, но слышали хорошее |
| **Grafana Loki** | Нет полнотекстового поиска; сложно масштабировать; хорош для алертинга по лейблам (как Prometheus для логов) |
| **Prometheus** | Используется для метрик из Kubernetes; Loki — его «аналог» для логов |

---

## 6. Kibana vs Grafana

| Аспект | Kibana | Grafana |
|--------|--------|---------|
| Полнотекстовый поиск | Да (основная сила) | Нет |
| Фильтры и drill-down | Отличные | Базовые |
| Визуализация трендов | Средняя (улучшается) | Отличная |
| Алертинг | Только в платной версии | Из коробки |
| Основное назначение | Поиск по логам | Дашборды и тренды |

---

## 7. Хранение логов

- **Retention:** ~1 месяц на hot-нодах, ещё ~2 недели на тёплых нодах.
- **Объём:** ~19 ТБ (без учёта снапшотов на S3).
- Архивные снапшоты — на S3 (бездонное хранилище, не считали).

## 8. Q&A — ключевые моменты

- **Почему Kafka, а не RabbitMQ?** — Больше экспертизы и документации по связке Logstash + Kafka.
- **Как собираете логи с K8s?** — Filebeat как DaemonSet на каждую ноду.
- **Loki vs Elasticsearch?** — Loki хорош, если не нужен полнотекстовый поиск и нужен алертинг по лейблам. Elasticsearch мощнее, но требует больше ресурсов и экспертизы.
- **Стоит ли менять стек в новой команде?** — Да, стоит рассмотреть все варианты и выбрать под задачу. Нет серебряной пули.

## Ключевые выводы

1. Даже в быстрорастущей команде с фокусом на продукт важно **заранее** думать о мониторинге.
2. ELK — проверенный стек, но при высоких нагрузках требует понимания внутренних механизмов (batch size Logstash, шарды ES, GC).
3. **Формула для ES:** 20 шардов на 1 ГБ heap — запомнить и соблюдать.
4. Logstash при highload: разделять входящие pipeline по топикам Kafka.
5. Grafana — для трендов и алертинга, Kibana — для поиска по логам. Оба инструмента дополняют друг друга.
6. Нет идеального инструмента — выбирать под конкретную задачу.
