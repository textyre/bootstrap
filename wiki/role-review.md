Каждый ответ на вопрос должен быть уникальным и не дублировать информацию из других вопросов. Недочеты должны быть уникальны. Точки роста должны быть уникальными. Если есть похожие вопросы, то это значит, что нужно дополнительно еще раз провести работу над вопросом и найти ответы в роли без дублирования информации.

# Common
1. Отвечает ли эта роль @wiki/standards/role-requirements.md ?
2. Какие недочеты существуют в этой роли? Информация не должна дублировать из информации к вопросу 1
3. Отсутствуют ли в этой роли антипатерны из @wiki/standards/role-requirements.md ? Информация не должна дублировать из информации к вопросу 1 и 2
4. Присутствуют ли бест практикс и советы в этой роли из @wiki/standards/role-requirements.md ? Информация не должна дублировать из информации к вопросу 1 и 2 и 3
5. Какие улучшения ты бы добавил в роль? 

# Зависимости
1. Роль избавлена от зависимостей?
2. Если в роли есть зависимости, они правильно установлены?

# Дублирование
1. Нет ли дублирования кода?
2. Нет ли одинаковых степов?
3. Есть ли части, которые можно вынести как common? 
4. Есть ли несколько шагов, которые можно сгруппировать и вынести в отдельный шаг задачу файл?

# Логирование
1. В этой роли понятное логирование действий?
2. Эта роль проверяет свои же шаги?
3. Логирование в отдельном файле?
5. Логи ротируются?
6. Logs — какие файлы, где, формат, как искать ошибки

# Тестирование
1. Тестирование покрывает все кейсы: основной, пограничный, ошибочный?
2. Тест кейсы описаны в ридми?
3. Testing — нет критериев успеха/провала

# README
1. README актуальное?
2. README полностью описывает то, что делает роль?
3. Роль соответствует тому, что описано в README?
4. Все шаги описаны в ридми?
5. Логирование описано в ридми?
6. Тестирование пайплайн описано в ридми?
7. Ридми отражает реальный пайплайн?
8. Execution Flow описан в ридми?
9. Troubleshooting описан в ридми?
10. Уровни безопасности переменных ?
11. File map — 20 файлов без карты "что трогать, что нет"
