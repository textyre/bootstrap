# Постмортем: архитектура задачи в Infrastructure as Code

**Дата:** 2026-05-08
**Категория:** Архитектура задачи
**Статус:** Draft
**Область:** Infrastructure as Code, архитектурные границы, decomposition, execution workflow, review readiness

## Итог

В ходе работы над IaC-задачей исполнитель допустил архитектурный drift: вместо того чтобы вывести решение из описания задачи, существующих проектных стандартов и фактов выполнения, он начал проектировать дополнительные механизмы, менять границы ответственности и усложнять orchestration flow.

Инцидент не был связан с одной конкретной строкой кода. Основная проблема была в архитектурном процессе: задача, подзадачи, code boundaries, execution workflow и acceptance criteria не были сначала сведены в единый контракт. Из-за этого реализация начала двигаться не от цели и ограничений проекта, а от локально правдоподобных, но не подтвержденных предположений исполнителя.

## Контекст

Проект является Infrastructure as Code системой. В таких проектах архитектура выражается не только в документах, но и в структуре playbook, ролей, task-файлов, workflow entrypoints, тестовых сценариев и CI matrix.

В работе участвовали несколько ролей:

- **Задача** задает ожидаемое поведение и границы изменения.
- **Подзадачи** должны быть атомарными кирпичами, каждый из которых имеет смысл только как часть общей стены.
- **Исполнитель** должен реализовывать только ту архитектуру, которая выведена из задачи, стандартов проекта и фактов.
- **Принимающий** проверяет, что решение соответствует цели, а не просто содержит рабочий код.
- **Ревьюер** проверяет не только diff, но и архитектурную согласованность: ответственность компонентов, читаемость workflow, тестируемость и отсутствие скрытых побочных моделей.

## Impact

Ошибки архитектуры привели к следующим последствиям:

- реализация начала двигаться по неверной модели;
- top-level workflow стал сложнее, чем требовала задача;
- появились дополнительные сущности, которые не были частью исходного архитектурного контракта;
- принимающий не мог понять, что именно запускается, где граница между шагами и почему это считается правильным решением;
- review readiness снизилась, потому что код начал требовать объяснения через внутренние термины исполнителя, а не через задачу;
- тестирование сначала доказывало не тот контракт или доказывало его недостаточно явно.

## Root Cause

Корневая причина: исполнитель начал проектировать архитектуру до того, как зафиксировал архитектурный контракт задачи.

Вместо последовательности:

```text
прочитать задачу -> извлечь контракт -> определить границы ответственности -> проверить проектные стандарты -> собрать факты -> реализовать минимальную архитектуру -> закрепить тестами
```

фактически возникла последовательность:

```text
частично понять проблему -> придумать механизм -> встроить механизм в workflow -> объяснять механизм постфактум -> исправлять после возражений принимающего
```

Это создало четыре ключевые архитектурные ошибки.

## Ошибка 1: изобретение архитектурных сущностей вне контракта задачи

### Что произошло

Исполнитель ввел дополнительные архитектурные сущности и названия, которых не было в задаче и которые не были выведены из стандартов проекта. Эти сущности начали восприниматься как часть решения, хотя изначально они были только внутренней гипотезой исполнителя.

### Почему это проблема

В IaC-проекте новые архитектурные сущности не являются нейтральными. Каждая новая сущность создает:

- новое состояние;
- новый execution path;
- новый failure mode;
- новую точку документации;
- новую поверхность для тестирования;
- новую когнитивную нагрузку для принимающего и ревьюера.

Если сущность не требуется задачей, она ухудшает архитектуру даже тогда, когда технически может работать.

### Причина

Исполнитель спутал локальную реализационную идею с архитектурным контрактом. Вместо того чтобы сначала спросить "какая минимальная модель следует из задачи?", он начал проектировать "какой механизм мог бы решить похожую проблему?".

### Что нужно делать иначе

Перед добавлением любой новой архитектурной сущности исполнитель должен ответить письменно:

| Вопрос | Gate |
|---|---|
| Эта сущность прямо указана в задаче? | Если нет, нужен отдельный justification. |
| Она следует из проектного стандарта? | Если нет, нельзя считать ее default path. |
| Без нее задача невыполнима? | Если нет, сущность не должна появляться. |
| Она уменьшает или увеличивает состояние workflow? | Увеличение состояния требует отдельного review. |
| Ее можно объяснить принимающему одной фразой через цель задачи? | Если нет, архитектура слишком сложная. |

## Ошибка 2: размещение orchestration logic внутри неправильного слоя

### Что произошло

Логика жизненного цикла задачи начала встраиваться в компоненты, которые должны были оставаться простыми и читаемыми. Вместо чистого top-level workflow появились условия, фазы, проверки исполнения и сообщения об ошибках внутри слоя, который должен был только описывать целевое состояние или выполнять свою узкую ответственность.

### Почему это проблема

В Infrastructure as Code важно различать:

- **orchestration layer**: порядок крупных шагов и границы между ними;
- **component layer**: декларативное управление конкретной частью системы;
- **validation layer**: проверка предпосылок и результата;
- **test layer**: доказательство, что контракт работает.

Когда orchestration logic попадает внутрь component layer, код становится "грязным":

- основной сценарий перестает читаться сверху вниз;
- компонент начинает знать слишком много о внешнем workflow;
- изменение одной подзадачи начинает влиять на unrelated roles/tasks;
- reviewer больше не может проверить ответственность компонента локально.

### Причина

Исполнитель пытался решить workflow-проблему через локальные условия и routing внутри существующего кода. Это произошло потому, что архитектурная граница между "порядком выполнения" и "настройкой компонента" не была зафиксирована до изменения файлов.

### Что нужно делать иначе

Перед изменением top-level IaC-файла нужно классифицировать изменение:

| Тип изменения | Правильное место |
|---|---|
| Порядок крупных lifecycle-шагов | Отдельный entrypoint, playbook или workflow task |
| Настройка компонента | Роль или task внутри роли |
| Проверка предпосылок роли | Preflight/validate task внутри роли |
| Проверка результата роли | Verify task или Molecule verify |
| Recovery/continuation behavior | Только после отдельного architectural approval |

Если изменение делает top-level файл менее читаемым, это сигнал остановиться и вынести ответственность в правильный слой.

## Ошибка 3: смешение разных ответственностей в одной подзадаче

### Что произошло

В одной архитектурной области были смешаны две разные ответственности: подготовка жизненного цикла системы и прикладная настройка системы. Из-за этого часть workflow начала выполняться не в том месте, где ее ожидает задача, а downstream-компоненты стали зависеть от состояния, которое еще не было стабилизировано.

### Почему это проблема

В IaC архитектуре похожие действия не всегда принадлежат одной ответственности. Например, "установить пакет", "обновить базу пакетов", "обновить систему", "установить declared application dependencies" и "подготовить runtime для downstream roles" могут использовать один и тот же package manager, но архитектурно быть разными операциями.

Если эти операции смешиваются:

- downstream-компоненты запускаются до завершения lifecycle boundary;
- tests начинают проверять side effects вместо явного контракта;
- reviewer не видит, где заканчивается подготовка системы и начинается настройка продукта;
- повторный запуск может иметь другое значение, чем первый запуск.

### Причина

Исполнитель классифицировал работу по техническому инструменту, а не по архитектурной ответственности. То есть вопрос был сформулирован как "какой модуль/роль умеет это делать?", а не как "какой компонент должен владеть этим решением?".

### Что нужно делать иначе

Подзадачи нужно разделять по ownership, а не по используемому инструменту:

| Вопрос | Правильный критерий |
|---|---|
| Это lifecycle preparation? | Отдельный lifecycle step. |
| Это declared configuration? | Component role. |
| Это dependency конкретного компонента? | Внутри owning component. |
| Это глобальная mutation системы? | Только в явно названном system-level step. |
| Это нужно всем downstream-компонентам до старта? | Это boundary, а не обычная role task. |

## Ошибка 4: stateful continuation вместо stateless workflow

### Что произошло

Исполнитель начал рассматривать продолжение старого execution context как способ решения задачи. Это привело к идеям сохранения checkpoint, resume-state, marker files или автоматического продолжения после boundary. При этом задача требовала не продолжить старый runtime, а начать новый независимый run после завершения boundary.

### Почему это проблема

Stateful continuation в IaC является дорогой архитектурной моделью. Она требует:

- persistent state format;
- rules для stale state;
- cleanup strategy;
- retry semantics;
- failure recovery;
- secret handling;
- test coverage для interrupted runs;
- clear ownership того, кто пишет и кто читает state.

Если задача может быть решена stateless workflow, stateful continuation является over-engineering и создает новые failure modes.

### Причина

Исполнитель пытался сохранить ход выполнения вместо того, чтобы признать boundary как естественную границу между двумя независимыми runs. Это произошло из-за неверного предположения, что один пользовательский workflow обязан быть одним непрерывным process-level execution.

### Что нужно делать иначе

Перед добавлением любого persistent state в IaC workflow нужно пройти decision gate:

| Вопрос | Требуемый ответ |
|---|---|
| Можно ли завершить текущий run и начать новый? | Если да, persistent state не нужен. |
| Нужно ли переносить runtime variables? | Если нет, checkpoint не нужен. |
| Можно ли пересобрать facts заново? | Если да, stale facts сохранять нельзя. |
| Что произойдет при изменении кода между runs? | Если ответ не определен, continuation запрещен. |
| Кто очищает устаревший state? | Если владельца нет, state запрещен. |

## Detection Gaps

Эти архитектурные ошибки не были пойманы сразу, потому что отсутствовали ранние gates:

| Gap | Что должно было сработать |
|---|---|
| Новые термины появились до архитектурного контракта | Glossary/contract review до кода |
| Top-level workflow усложнился | Readability review до merge |
| Подзадачи смешали ответственности | Ownership mapping перед implementation |
| Stateful model появилась без approval | Persistent state decision gate |
| Тесты сначала не доказывали нужный контракт | Acceptance criteria должны быть связаны с tests до реализации |

## Что сработало

Несмотря на ошибки, процесс был восстановлен после того, как принимающий остановил неверную архитектуру и потребовал:

- описать архитектуру до дальнейшей реализации;
- убрать лишние сущности;
- вернуть простую модель;
- собрать факты до утверждений;
- закрепить решение тестами;
- фиксировать checkpoints в основном артефакте задачи.

## Preventive Rules

### Для исполнителя

1. Не вводить новые архитектурные сущности без явной связи с задачей.
2. Сначала писать architecture contract, затем менять код.
3. Разделять lifecycle orchestration и component configuration.
4. Не добавлять persistent state, если можно начать новый run.
5. Не считать технически похожие операции одной ответственностью.
6. Объяснять top-level workflow через задачу, а не через внутренние названия исполнителя.

### Для принимающего

1. Останавливать работу, если появляются новые сущности без justification.
2. Требовать карту ownership до реализации.
3. Проверять, что подзадачи атомарны и не имеют смысла вне общей стены.
4. Не принимать тесты, которые доказывают не тот архитектурный контракт.

### Для ревьюера

1. Проверять, что top-level workflow читается как process map.
2. Проверять, что роли не владеют чужими lifecycle responsibilities.
3. Идентифицировать stateful continuation как high-risk design.
4. Требовать explicit test coverage для каждого architecture boundary.

## Критерий, что проблема не повторяется

Похожая задача считается архитектурно готовой к реализации только если до изменения кода можно ответить:

```text
1. Какой минимальный workflow требует задача?
2. Какие компоненты владеют какими ответственностями?
3. Где проходит lifecycle boundary?
4. Есть ли persistent state? Если да, почему без него нельзя?
5. Какой тест доказывает каждый boundary?
6. Может ли принимающий понять workflow без внутренних терминов исполнителя?
```

Если хотя бы один ответ отсутствует, задача не готова к реализации.
