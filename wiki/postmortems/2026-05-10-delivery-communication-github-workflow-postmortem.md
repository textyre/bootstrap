# Постмортем: коммуникация, GitHub delivery и закрытие задач

**Дата:** 2026-05-10
**Категория:** коммуникация, GitHub issue/PR, CI, Git workflow, project management
**Статус:** draft
**Назначение:** практическая база для будущих скиллов, runbook-инструкций и проектной документации.

## Краткий итог

Во время работы над задачей несколько раз ломался не сам кодовый путь, а delivery-контур вокруг него: где фиксируется решение, как обновляется GitHub issue, как объясняется статус PR, как интерпретируется CI, как называются ветки, как закрывается issue и как переводятся project items.

Главная проблема: агент вел работу как набор локальных действий и сообщений в чат, а не как управляемый процесс с одним главным артефактом, явными checkpoint-ами, проверяемыми фактами и обязательным закрывающим протоколом.

Этот постмортем объединяет четыре категории ошибок:

- Коммуникация и GitHub issue.
- CI и GitHub Actions.
- Git workflow.
- Project management и закрытие задачи.

## Практическая цель документа

Документ должен быть пригоден для последующего перевода в:

- скилл для работы с GitHub issue;
- скилл для delivery через PR;
- runbook для чтения CI и GitHub Actions;
- checklist для merge/close/project-status flow;
- инструкцию для агентов, которые работают с задачами без постоянного подтверждения человека.

Фокус документа не на том, какая именно роль, плейбук или сценарий менялись. Фокус на повторяемом рабочем процессе: как не потерять контекст, не исказить статус, не создать лишние артефакты и не оставить задачу в полузакрытом состоянии.

## Целевое рабочее состояние

| Этап | Главный артефакт | Что обязан сделать агент | Признак готовности |
| --- | --- | --- | --- |
| Разбор задачи | GitHub issue body | Обновить описание задачи, критерии приемки, план и checkpoints в самом issue body, если это требуется задачей | Issue body содержит актуальную задачу без необходимости читать чат |
| План работ | Issue checklist | Разложить работу на атомарные пункты, которые можно проверять последовательно | Каждый пункт имеет понятный результат и условие перехода дальше |
| Реализация | Git branch + commits | Держать scope ветки чистым, не смешивать unrelated изменения | `git diff --name-only` соответствует задаче |
| Проверка | PR checks + логи CI | Разделять прямые проверки задачи, побочные проверки и инфраструктурные сбои | В отчете ясно, что проверено, что упало и относится ли это к задаче |
| Ревью и merge | PR | Проверить, что PR закрывает нужную issue или закрыть ее вручную после merge | PR merged, issue closed/completed |
| Project management | GitHub Projects | Проверить отдельно статус issue item и PR item | Оба элемента в нужном проектном статусе |
| Финальный отчет | Issue/PR + чат | Дать короткий отчет со ссылками на факты | Человек может принять результат без восстановления контекста вручную |

## Влияние

Ошибки delivery-контура увеличили стоимость ревью и приемки:

- Принимающий не мог понять, где находится актуальное описание задачи: в чате, issue body или комментарии.
- Возникла путаница между тем, что было сделано в коде, что было проверено, что упало в CI и что относится к задаче.
- Несколько раз приходилось уточнять базовые вещи: что именно запускалось, на какой стадии задача, почему выбран такой workflow, что является фактом, а что выводом.
- После merge задача не была автоматически завершена полностью: issue и project item потребовали отдельной проверки и ручного доведения до Done.
- Часть времени была потрачена не на качество решения, а на восстановление доверия к процессу.

## Первопричины

Первопричина была не в одном инструменте. Она состояла из нескольких процессных разрывов.

| Разрыв | Что пошло не так | Почему это опасно |
| --- | --- | --- |
| Artifact ownership | Агент не удерживал один главный артефакт задачи | Описание задачи расползается между чатом, комментариями и локальными выводами |
| Delivery state model | Агент не разделял состояния issue, PR, branch, CI и project item | Можно считать задачу завершенной, хотя один из контуров еще не закрыт |
| Evidence discipline | Агент смешивал выводы, предположения и факты | Ревьюер не может принять решение по непроверяемым утверждениям |
| Tool fallback discipline | Агент не всегда явно отделял сбой инструмента от сбоя продукта | Ошибка `gh` или GitHub API может быть ошибочно воспринята как ошибка задачи |
| Scope hygiene | Агент не всегда достаточно рано фиксировал границы изменений | В PR попадают unrelated изменения или непонятные side effects |

## Ошибка 1. Неверный главный артефакт коммуникации

### Что произошло

Часть содержания, которое должно было жить в GitHub issue body, сначала оказалось в чате или отдельном GitHub comment. При этом задача была сформулирована на русском, а один из промежуточных комментариев был добавлен на английском.

### Почему это ошибка

Если пользователь просит работать с GitHub issue, главным артефактом является issue. Чат может быть рабочим каналом, но не должен становиться единственным местом, где существует план, критерии приемки, checkpoints или блокеры.

Отдельный комментарий тоже не равен описанию задачи. Комментарий полезен для обсуждения, но если меняется постановка, порядок работ, criteria или task contract, это должно попадать в issue body.

### Как это проявилось в работе

Ревьюер справедливо указал, что задача должна быть описана в issue, а не в ответе чата или отдельном комментарии. Также был отдельно отмечен языковой разрыв: задача на русском, значит описание, checklist и комментарии к постановке должны быть на русском, если пользователь не просил другое.

### Правильный практический паттерн

| Ситуация | Куда писать | Что писать |
| --- | --- | --- |
| Меняется постановка задачи | Issue body | Новое описание, контекст, ограничения, критерии приемки |
| Добавляются checkpoints | Issue body checklist | Атомарные шаги с проверяемым результатом |
| Нужно сообщить ход работы | Issue comment или чат | Короткий статус без изменения task contract |
| Нужно приложить доказательство | Issue comment или PR comment | Команда, вывод, ссылка на лог |
| Нужно объяснить итог | PR description, issue closing comment или чат | Что сделано, что проверено, что осталось |

### Правило для будущего скилла

Перед изменением GitHub issue агент обязан классифицировать действие:

- `body update`: меняет саму постановку, план, критерии, checklist.
- `comment`: добавляет доказательство, статус или обсуждение.
- `PR description`: объясняет реализацию и проверки для ревью.
- `chat`: кратко синхронизирует человека, но не заменяет GitHub artifact.

Если пользователь говорит “это должно быть в issue”, агент должен обновлять issue body, а не писать новый комментарий.

## Ошибка 2. Необъясненные термины и локальные ярлыки

### Что произошло

В отчетах появились термины вроде `Workstation Resume`, `result records`, `check mode`, `Phase B cleanup`, `preflight`. Часть терминов была не из исходной задачи, не из issue и не из принятой архитектуры. Они выглядели как самостоятельные сущности, хотя должны были быть либо внутренними техническими деталями, либо вообще не должны были появляться.

### Почему это ошибка

Новые термины создают ложную архитектуру. Ревьюер начинает выяснять не только “правильно ли решена задача”, но и “что вообще означает этот новый слой”.

Если термин не нужен пользователю для принятия решения, его лучше не вводить. Если термин нужен, его нужно определить до использования.

### Как это проявилось в работе

Пользователь несколько раз возвращал разговор к простой модели: есть две фазы, между ними reboot, после reboot продолжается основная настройка. Дополнительный термин `Resume` воспринимался как новая сущность, которую никто не просил создавать.

### Правильный практический паттерн

| Тип термина | Можно использовать? | Условие |
| --- | --- | --- |
| Термин из issue | Да | Использовать в том же смысле |
| Термин из кода | Да | Дать ссылку на файл или объяснить один раз |
| Внутренний термин агента | Нежелательно | Только если он помогает, и обязательно определить |
| Новый архитектурный термин | Нет без согласования | Сначала объяснить, зачем он нужен |

### Правило для будущего скилла

Перед финальным или промежуточным отчетом агент должен проверять словарь:

- Все ключевые термины взяты из issue, кода или документации.
- Если термин новый, рядом есть короткое определение.
- Если термин не нужен для решения, он удален из отчета.
- Нельзя заменять простую пользовательскую модель новой терминологией без явной пользы.

## Ошибка 3. Неразделенные типы CI-состояний

### Что произошло

Во время проверки PR смешивались разные типы CI-состояний: прямые проверки по задаче, матричные jobs, unrelated/sidecar checks, pending jobs, инфраструктурные ошибки GitHub CLI и реальные failures.

Отдельный пример: `gh pr checks --watch` вернул `504 Gateway Timeout`. Это ошибка наблюдения через инструмент или GitHub API, а не автоматически ошибка кода. Такой сбой нужно отдельно классифицировать.

### Почему это ошибка

CI является источником фактов только при правильной интерпретации. Если агент говорит “CI упал”, он обязан объяснить:

- какая именно проверка упала;
- относится ли она к измененной области;
- это failure продукта, теста, матрицы, окружения или GitHub API;
- какие логи это подтверждают;
- можно ли принимать PR при наличии unrelated/pending checks.

Без этой классификации ревьюер не понимает, нужно ли чинить код, инфраструктуру, workflow или просто повторить запрос.

### Как это проявилось в работе

В процессе были ситуации, где:

- check matrix падала из-за пустого или невалидного набора changed roles;
- часть jobs относилась к ролям или проверкам вне основного изменения;
- in-progress jobs еще не имели финальных логов;
- `gh pr checks --watch` не был надежным единственным способом наблюдения из-за `504 Gateway Timeout`;
- требовалось дополнительно объяснять, какие проверки реально подтверждают задачу.

### Правильная классификация CI

| Тип состояния | Что означает | Что делать агенту |
| --- | --- | --- |
| Прямой успех | Проверка напрямую покрывает измененную область и прошла | Отметить как evidence |
| Прямое падение | Проверка напрямую покрывает измененную область и упала | Читать логи, чинить или блокировать merge |
| Побочное падение | Упала проверка рядом с задачей, но не доказывает failure изменения | Отделить от direct scope и объяснить риск |
| Инфраструктурный или tooling-сбой | Сбой GitHub API, CLI, runner startup, timeout наблюдения | Повторить snapshot-запрос, не выдавать за failure кода |
| Pending | Проверка еще идет | Не делать финальный вывод, дать промежуточный статус |
| Skipped | Проверка пропущена по условиям workflow | Проверить, ожидаем ли skip |
| Empty matrix | Матрица не содержит jobs | Workflow должен явно и корректно обрабатывать empty case |

### Правило для будущего скилла

Любой CI-отчет должен иметь минимум четыре поля:

| Поле | Обязательный смысл |
| --- | --- |
| `check` | Название проверки или job |
| `state` | passed, failed, pending, skipped, infra-error |
| `scope` | direct, sidecar, unrelated, unknown |
| `evidence` | ссылка на run/log или команда с выводом |

Нельзя писать “CI упал” без `check`, `scope` и `evidence`.

## Ошибка 4. Неполный резервный путь при сбоях GitHub-инструментов

### Что произошло

Когда GitHub CLI или API давал нестабильный результат, например `504 Gateway Timeout`, процесс наблюдения мог выглядеть как тупик или как противоречивый статус. При этом существовали альтернативные способы получить состояние: повторный snapshot, просмотр конкретного workflow run, чтение logs после завершения job, GraphQL/REST fallback.

### Почему это ошибка

Инструмент наблюдения не является самой системой. Если `gh pr checks --watch` не отвечает, это не значит, что CI остановился, PR сломан или задача заблокирована. Это значит, что сломан конкретный канал наблюдения.

### Правильный practical fallback

| Сбой | Первый fallback | Второй fallback | Когда блокировать |
| --- | --- | --- | --- |
| `gh pr checks --watch` timeout | `gh pr checks` snapshot | `gh run list` по branch/PR | Только если невозможно получить состояние никаким путем |
| Logs unavailable while job running | Проверить job status | Дождаться completion и читать logs | Не блокировать до финального состояния |
| API returns partial data | Повторить запрос | Сверить через web/API другой endpoint | Блокировать только при расхождении критичных данных |
| Matrix job missing | Проверить workflow inputs/changed files | Проверить generated matrix output | Блокировать, если workflow не покрывает измененную область |

### Правило для будущего скилла

При сбое GitHub tooling агент должен явно писать:

- какой инструмент дал сбой;
- какая команда или endpoint использовались;
- какой fallback выбран;
- какой факт удалось подтвердить после fallback;
- остается ли задача заблокированной или это только временная проблема наблюдения.

## Ошибка 5. Имя ветки и чистота scope

### Что произошло

Название ветки было неверным несколько раз, пока пользователь не подтвердил корректное. Также возникал вопрос “а остальные изменения это что?”, потому что в рабочем дереве были изменения не только по главному ожидаемому фрагменту.

### Почему это ошибка

Branch name, staged files и PR scope являются частью task contract. Если они не совпадают с ожиданием, reviewer вынужден проверять не только качество кода, но и то, не уехал ли агент в чужой scope.

Неверное имя ветки само по себе не ломает код, но ломает доверие к workflow: если агент не удержал простое внешнее имя, возникает разумный вопрос, удержал ли он более сложные ограничения задачи.

### Правильный practical pattern перед commit/push

| Проверка | Команда | Что должно быть ясно |
| --- | --- | --- |
| Текущая ветка | `git branch --show-current` | Имя соответствует задаче и указанию пользователя |
| Измененные файлы | `git status --short` | Нет неожиданных файлов |
| Scope diff | `git diff --name-only` | Каждый файл объясним задачей |
| Staged scope | `git diff --cached --name-only` | В commit попадет только нужное |
| Commit message | `git log -1 --oneline` после commit | Сообщение отражает задачу |
| Remote branch | `git status --short --branch` | Tracking не указывает на удаленную ветку после cleanup |

### Правило для будущего скилла

Перед `git push` агент обязан выполнить branch/scope gate:

- Название ветки сверено с explicit request или issue title.
- Все измененные файлы перечислены и классифицированы как in-scope.
- Unrelated изменения не staged.
- Если есть сомнительные файлы, агент останавливается и объясняет scope.
- Push выполняется только после чистого scope gate.

## Ошибка 6. Смешивание готовности кода, PR и задачи

### Что произошло

После merge PR задача еще не была полностью закрыта. Нужно было отдельно проверить issue state и project item state. В частности, PR merge не гарантировал автоматическое закрытие issue, потому что связь closing reference могла отсутствовать или быть недостаточной.

### Почему это ошибка

В GitHub delivery есть несколько разных “готово”:

- code done: изменения реализованы локально;
- tests done: проверки пройдены;
- PR done: pull request merged;
- issue done: GitHub issue закрыта как completed;
- project done: issue и PR project items переведены в Done;
- user done: пользователь получил понятный отчет.

Если считать merge финалом, можно оставить задачу открытой в issue tracker или project board.

### Правильная модель состояний

| Состояние | Проверка | Что делать, если не готово |
| --- | --- | --- |
| Branch pushed | Remote branch существует | Push или исправить remote |
| PR open | PR создан и связан с branch | Создать/обновить PR |
| Checks complete | Required/direct checks passed или классифицированы | Дождаться, починить или явно зафиксировать риск |
| PR merged | PR state `MERGED` | Merge или объяснить блокер |
| Issue closed | Issue state `CLOSED`, reason `COMPLETED` | Закрыть вручную, если auto-close не сработал |
| PR project done | PR project item status `Done` | Перевести project item |
| Issue project done | Issue project item status `Done` | Перевести project item |

### Правило для будущего скилла

`Merged` не равно `Done`. После merge агент обязан выполнить closure gate:

- Проверить PR state.
- Проверить issue state.
- Проверить issue close reason.
- Проверить project status issue item.
- Проверить project status PR item, если PR тоже добавлен в проект.
- Только после этого писать, что задача закрыта.

## Ошибка 7. Недостаточно явные checkpoint-и для автономной работы

### Что произошло

Пользователь просил вести последовательный процесс и отмечать checkpoints в issue по мере выполнения. В такой модели агент не должен ждать подтверждения на каждый шаг, но обязан оставлять следы прогресса там, где они нужны принимающему.

Риск заключался в том, что checkpoints могли стать просто текстовым списком без доказательств или могли жить только в чате.

### Почему это ошибка

Автономная работа без checkpoints превращает ревью в forensic-анализ. Принимающий должен заново устанавливать:

- что агент собирался сделать;
- что реально сделал;
- где остановился;
- какие проверки прошли;
- какие проверки не запускались;
- какие решения были изменены по ходу.

### Правильный checkpoint protocol

| Checkpoint type | Где фиксировать | Содержимое |
| --- | --- | --- |
| Plan checkpoint | Issue body | Что будет сделано и в каком порядке |
| Implementation checkpoint | Issue checklist или PR | Какой пункт выполнен, какие файлы изменены |
| Verification checkpoint | Issue/PR comment | Команда, вывод, ссылка на CI/log |
| Blocker checkpoint | Issue comment | Что блокирует, почему, что нужно от человека |
| Closure checkpoint | Issue/PR | Merge, issue close, project done |

### Правило для будущего скилла

Checkpoint считается валидным только если содержит:

- конкретное действие;
- конкретный результат;
- evidence или ссылку на место, где evidence можно проверить;
- следующий шаг или статус “больше действий не требуется”.

## Ошибка 8. Недостаточно строгая граница между сообщением человеку и фактом системы

### Что произошло

В ходе работы пользователь несколько раз указывал, что ответы содержали противоречивые или непроверенные утверждения. В delivery-контуре это особенно опасно: если агент сообщает неверный статус CI, branch, issue или project item, человек может принять неправильное решение.

### Почему это ошибка

Любой статус delivery должен быть воспроизводимым. Нельзя говорить “прошло”, “закрыто”, “упало”, “готово”, если рядом нет источника факта.

### Практическое правило статусов

| Нельзя писать | Нужно писать |
| --- | --- |
| “CI упал” | “Job `X` в run `Y` завершился `failed`; scope direct; лог показывает `Z`” |
| “Задача закрыта” | “Issue `#N` state `CLOSED`, reason `COMPLETED`; project item status `Done`” |
| “Ветка удалена” | “Remote branch удалена; local branch удалена; `git status --branch` показывает `master...origin/master`” |
| “Проверки прошли” | “Checks `A`, `B`, `C` passed; check `D` skipped по условию `E`” |
| “Это unrelated” | “Файл/job относится к роли/пути `X`, который не менялся в PR; direct scope: `Y`” |

### Правило для будущего скилла

Если статус влияет на решение человека, он должен быть оформлен как evidence-backed statement:

`claim -> source -> exact observed value -> implication`.

Пример:

`PR merged -> GitHub PR state -> MERGED -> можно переходить к issue/project closure gate`.

## Практический runbook: обновление GitHub issue

Использовать, когда задача живет в GitHub issue и пользователь просит обновить постановку, план или checkpoints.

| Шаг | Действие | Условие остановки |
| --- | --- | --- |
| 1 | Прочитать issue body, labels, comments и текущие project links | Невозможно получить issue через GitHub |
| 2 | Определить язык issue и писать на том же языке | Язык неоднозначен и пользователь явно просил другой |
| 3 | Разделить body update и comment update | Непонятно, меняется постановка или добавляется статус |
| 4 | Обновить issue body, если меняются договоренность по задаче, план, критерии приемки или checklist | Нет прав на редактирование issue |
| 5 | Добавить comment только для evidence/status, если это не меняет task contract | Комментарий заменяет body update |
| 6 | После обновления перечитать issue и подтвердить, что нужный текст находится именно там | Issue body не обновился |

## Практический runbook: отчет по PR и CI

Использовать при проверке PR, особенно если CI частично падает или GitHub CLI ведет себя нестабильно.

| Шаг | Действие | Условие остановки |
| --- | --- | --- |
| 1 | Получить PR number, branch, commit SHA | PR не найден |
| 2 | Получить список checks snapshot-командой | GitHub CLI/API недоступны |
| 3 | Классифицировать каждый check как direct, sidecar, unrelated или unknown | Нельзя определить scope |
| 4 | Для failed direct check прочитать logs | Logs недоступны после completion |
| 5 | Для pending check не делать финальный вывод | Пользователь просит финальный merge без завершения required checks |
| 6 | Для tooling error применить fallback, а не считать это failure кода | Все fallback-каналы недоступны |
| 7 | Отчитаться таблицей `check/state/scope/evidence` | Нет evidence для важного claim |

## Практический runbook: чистота ветки и коммита

Использовать перед commit, push, PR creation и merge.

| Шаг | Действие | Условие остановки |
| --- | --- | --- |
| 1 | Проверить текущую ветку | Ветка не соответствует explicit request |
| 2 | Проверить `git status --short` | Есть неожиданные modified/untracked файлы |
| 3 | Проверить `git diff --name-only` | Есть файлы вне scope |
| 4 | Stage только intended files | Невозможно отделить изменения |
| 5 | Проверить staged diff | Staged diff содержит unrelated изменения |
| 6 | Commit с сообщением по задаче | Commit message не отражает change |
| 7 | Push только после branch/scope gate | Remote отклоняет push или branch name неверный |

## Практический runbook: merge, закрытие issue и project Done

Использовать после успешного merge или когда пользователь говорит закрыть задачу.

| Шаг | Действие | Условие остановки |
| --- | --- | --- |
| 1 | Проверить PR merged state | PR не merged |
| 2 | Проверить, закрылся ли issue автоматически | Issue все еще open |
| 3 | Если issue open, закрыть вручную с reason completed | Нет прав или issue не та |
| 4 | Проверить issue state и close reason | State не `CLOSED` или reason не completed |
| 5 | Проверить issue project item status | Item отсутствует или status не Done |
| 6 | Проверить PR project item status, если он есть | PR item status не Done |
| 7 | Выполнить local git cleanup | Локальная ветка все еще tracking deleted remote |
| 8 | Дать финальный отчет с фактами | Не хватает evidence по одному из closure states |

## Контрольные точки

Эти gates должны стать обязательными stop/check точками для будущих скиллов.

| Gate | Когда | Что проверяет | Если не проходит |
| --- | --- | --- | --- |
| Artifact gate | Перед обновлением GitHub | Правильное место изменения: body/comment/PR/chat | Не писать, пока не выбран артефакт |
| Language gate | Перед публикацией текста | Язык совпадает с issue/user request | Переписать на нужном языке |
| Terminology gate | Перед отчетом | Нет новых неочевидных терминов без определения | Определить или удалить термин |
| Scope gate | Перед commit/push | Все файлы in-scope | Остановиться и объяснить diff |
| CI gate | Перед merge/report | Checks классифицированы по state/scope/evidence | Не делать финальный вывод |
| Closure gate | После merge | Issue и project item доведены до Done | Закрыть/перевести или сообщить blocker |

## Кандидаты на будущие скиллы

Этот постмортем можно разложить на следующие практические скиллы.

| Skill | Назначение | Основные rules |
| --- | --- | --- |
| `github-issue-steward` | Вести GitHub issue как главный task artifact | Body-first updates, language matching, checkpoint protocol |
| `pr-ci-triage` | Классифицировать PR checks и CI failures | Direct/sidecar/unrelated, tooling fallback, evidence table |
| `git-scope-gate` | Проверять ветку, staged files и diff перед push | Branch name, file scope, staged-only intended changes |
| `delivery-closure` | Доводить merge до закрытой задачи | PR merged, issue completed, project items Done |
| `evidence-backed-reporting` | Писать статусы без неподтвержденных claim-ов | Claim/source/value/implication format |

## Что делать иначе в будущем

| Ситуация | Старое ошибочное поведение | Новое обязательное поведение |
| --- | --- | --- |
| Нужно переписать задачу | Написать план в чат или comment | Обновить issue body на языке задачи |
| Нужно объяснить прогресс | Использовать новые термины без определения | Использовать словарь issue или определить термин |
| CI дал непонятный статус | Сказать “CI упал” | Разложить по check/state/scope/evidence |
| GitHub CLI упал по timeout | Считать это blocker задачи | Классифицировать как tooling issue и применить fallback |
| Перед push есть extra files | Протащить их в общий diff | Остановиться на scope gate |
| PR merged | Считать задачу готовой | Выполнить closure gate для issue и project |

## Критерии приемки нового процесса

Будущий агент работает правильно, если:

- Актуальная постановка задачи находится в GitHub issue body.
- Checkpoints задачи видны в issue или PR, а не только в чате.
- Все публичные тексты написаны на языке исходной задачи, если пользователь не просил иначе.
- CI-отчет разделяет direct, sidecar, unrelated и infra/tooling states.
- Каждый важный delivery-claim имеет источник факта.
- Перед commit/push выполнен scope gate.
- После merge выполнен closure gate.
- Issue закрыта как completed.
- Project item issue переведен в Done.
- PR project item переведен в Done, если он существует.

## Дальнейшие действия

| Action | Owner | Result |
| --- | --- | --- |
| Создать скилл для GitHub issue body-first workflow | Agents/process | Агент не пишет task contract в комментарии вместо body |
| Создать CI triage checklist | Agents/process | Любой CI failure классифицируется по scope и evidence |
| Создать git scope gate checklist | Agents/process | Перед push нет неожиданных файлов и неверных веток |
| Создать closure gate checklist | Agents/process | Merge не считается Done без issue/project verification |
| Добавить шаблон финального delivery-отчета | Agents/process | Человек видит status/evidence/remaining risk без уточнений |

## Финальный вывод

Задача не считается завершенной только потому, что код написан или PR merged. Для управляемой инфраструктурной работы нужен полный delivery loop:

`issue body -> checkpoints -> scoped branch -> PR -> CI classification -> merge -> issue completed -> project Done -> evidence-backed report`.

Главное изменение на будущее: агент должен вести delivery как проверяемую систему состояний, а не как поток сообщений. Тогда ревьюер видит не “что агент сказал”, а “что система подтвердила”.
